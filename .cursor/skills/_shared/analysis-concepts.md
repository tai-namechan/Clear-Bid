# 共通分析概念（全スキル共有リファレンス）

bugfix / incident / perf-review / review-only / spec / clear-bid-diagnosis が参照する共通定義。

---

## Data Cardinality（データ件数前提）

対象データが「0/1件・複数件・大量件」のどれかを、**推測ではなく取得コードを根拠に**確定する。

判定手順:
1. page / composable → api → domain / repository まで遡る
2. limit / 単体取得 / 全件取得かを確認する
3. 所有者・期間などの条件から現実的な件数を見積もる

| 区分 | 目安 | 扱い |
|---|---|---|
| 0/1件 | id 指定 | ループを気にしなくてよい |
| 複数件 | 〜100件 | 一括処理可 |
| 大量件 | 1000件超になり得る | 分割必須（`db-memory-performance.mdc`） |

## Filter Location（フィルタの実行場所）

絞り込みが **永続化層 / API クエリ** か **メモリ上の filter** かを特定する。

- 原則: 取得側で絞る。SQL/クエリ済みをクライアント全件 filter に置き換える変更は危険信号

## Ops Delta（処理量の増減）

変更前後で以下の増減を数で示す。

- クエリ / `$fetch` 回数
- ループ回数
- メモリに載るレコード数

## 確信度（Confidence）

**80% 未満ならパッチ提案禁止**（調査項目のみ）。

| 確信度 | 条件 |
|---|---|
| 90%〜 | 原因コード行を特定し、再現メカニズムを実コードで説明できる |
| 80%〜 | 原因行は特定したが再現未確認 |
| 〜79% | 複数候補 / 推測ベース → **パッチ禁止** |

自己検証には「確信度% / 根拠 / 不確実点 / 追加確認事項」を含める。

## 地雷チェック（危険操作リスト）

NG が1つでもあればパッチではなく提案のみ。

| # | チェック項目 | 根拠ルール |
|---|---|---|
| 1 | ボトムナビ/共通レイアウトを不用意に変更していないか | `global-workflow.mdc` |
| 2 | デザイントークン/共通 UI クラスを破壊していないか | `design-consistency.mdc` |
| 3 | 所有者スコープ漏れがないか | `db-memory-performance.mdc` |
| 4 | 大量データの全件取得・N+1 を生んでいないか | `db-memory-performance.mdc` |
| 5 | テストが外部 HTTP を投げる可能性がないか | `http-safety.mdc` |
| 6 | 金額計算や判断候補を AI に任せていないか | `clear-bid-domain.mdc` |
| 7 | AI 推定だけで BLOCK にしていないか | `clear-bid-domain.mdc` |
| 8 | 総合点/スコアバーを追加していないか | `clear-bid-domain.mdc` |
| 9 | エラー握りつぶしでテストを通していないか | `test-quality-standards.mdc` |
| 10 | 差分最小か（整形・ついでリファクタがないか） | `global-workflow.mdc` |

## SEARCH_SCOPE の規律

- 指定範囲を主に探索する。外に原因があると判断したら勝手に広げず報告する
- 未指定なら着手前に対象範囲を提示する

## 出力の共通ルール

- 報告系は `report-output-format.mdc` に従い、1つの markdown コードブロックで囲む
- ネストしたコードブロックを書かない
